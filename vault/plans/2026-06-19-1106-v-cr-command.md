# v-cr-command — team plan

Design + build a new framework command `/v-cr`: auto-detect the forge (GitHub / GitLab /
Bitbucket+Jira), deploy a tool-grounded critic swarm against a PR/MR, and post grounded review
comments back to the forge — context-aware from vault + the PR/MR description + the linked task.
Grounded in research on how CodeRabbit, Greptile, Qodo, Copilot, Graphite et al. do automated CR
(2025–2026), and revised by a 3-critic design panel (Architect, Skeptic, Security).

## Task
Build `/v-cr` — forge-auto-detecting, vault- and task-aware, agent-swarm code review that posts
inline + summary comments to GitHub/GitLab/Bitbucket. Keywords: code-review, source-detection,
pr-comments, agent-swarm, context-aware, task-grounding.

## Research digest — what other companies do (de-facto 2026 architecture)
The serious tools (CodeRabbit, Greptile, Qodo Merge/PR-Agent, GitHub Copilot review, Graphite Agent,
Cursor Bugbot, Sourcery, Bito, Ellipsis, Korbit, Codacy, Gemini Code Assist) converged on one design:
1. **Trigger duality** — auto-on-PR (webhook) + on-demand command (`@bot review`) + CI step.
2. **Feedback triad** — inline diff comments + a structured PR summary/walkthrough + one-click
   committable fixes; status check usually opt-in, default non-blocking comment-only.
3. **Context beyond the diff is table stakes** — (a) whole-repo grounding via code-graph / RAG; (b)
   config-as-code natural-language rule files (`.coderabbit.yaml`, `greptile.json`, `CLAUDE.md`,
   `AGENTS.md`); (c) linked issue/ticket context (Jira/Linear/GitHub).
4. **Learned org memory** — persists team preferences, suppresses repeated nits; security/logic exempt.
5. **Multi-agent / multi-step pipeline, generate-then-verify** — the noise-control engine is
   *separating comment generation from comment evaluation* (CodeRabbit verification agent, Ellipsis
   ConfidenceFilter, Sourcery validation pass, Greptile independent evaluator).
6. **Deterministic static analysis under the LLM** — linters/SAST as the high-precision floor.
7. **Noise control is THE product axis** — confidence thresholds ("say nothing when unsure" — Copilot
   stays silent in 29% of reviews), severity levels, volume caps, learned suppression. Market is
   precision-first (Graphite <5% negative-comment rate, Cursor "only real bugs").
8. **Platform reach segments the market** — broadest (GH+GL+BB+Azure) = CodeRabbit & Qodo.

**Key insight:** the generate-then-verify panel is what `/v-team`'s diff-review already does — BUT
v-team's loop *fixes code and re-rounds*, which is meaningless against a static remote PR. So v-cr does
**not** reuse v-team's loop; it reuses the *sub-procedure* (analyzers-first → parallel read-only
decorrelated critics → de-biased synthesis → grounding-rule confidence filter), extracted into a
shared **single-pass** module, plus forge adapters for fetch+post and task-context ingestion.

## Converged plan (v1 — after panel)

### Positioning
- `/v-work`, `/v-team` = **author** code (propose → build).
- `/v-cr` = **review** existing code on a PR: fetch diff → critique → post comments. **Read-only on the
  codebase, write-only to the forge (comments). INVARIANT: never commits, pushes, or applies code**
  (sec-6). Does not depend on the v-work/v-team *lifecycles*; depends only on three shared assets:
  `personas/_resolution.md`, the new `commands/_shared/critic-panel.md`, and a context loader (arch-3,
  skeptic-9).

### Single-pass pipeline (thin dispatcher + 5 steps + a shared panel module)

**1. DETECT** (`steps/01-detect.md`) — resolve forge + PR/MR; security-hardened.
   - Parse the upstream remote URL: HTTPS + scp-like SSH (no-slash-before-colon rule); resolve
     `~/.ssh/config` Host aliases via `ssh -G <host>` before giving up (skeptic-7). Strip `user@`/`.git`.
   - **Host allowlist + strict validation** (sec-3): exact-match against known forge domains
     (github.com / gitlab.com / bitbucket.org) + a **user/global** self-hosted host map (NOT read from
     repo-controlled files). Reject look-alike hosts (`github.com.evil.test`), IP-literals, and non-TLS.
     A self-hosted host requires explicit one-time user confirmation before first use.
   - **Decouple PR identity from the local checkout** (arch-2, skeptic-6): resolve the PR's base repo
     identity from the forge API, not the local tree. `gh pr view` / `glab mr view` / Bitbucket REST
     `q=source.branch`. Accept explicit `/v-cr <url|number>` override (the escape hatch).
   - Emit: adapter (`github`|`gitlab`|`bitbucket-cloud`|`bitbucket-server`), base `owner/repo`, PR id,
     resolved target string `host/owner/repo#PR`, and whether the PR head is a **fork / public** (drives
     the egress policy in step 3/4).

**2. GATHER CONTEXT** (`steps/02-gather.md`) — the "context-aware" requirement; untrusted-input aware.
   - Forge: PR/MR diff, title, body, linked issues (`closingIssuesReferences` / `closes_issues`).
   - **Secret-scan the diff BEFORE it enters any LLM context** (sec-2): gitleaks/trufflehog-style rules
     + token-shape regex (`gh[pousr]_`, `glpat-`, `Bearer `, `ATATT`, `xox…`, AWS keys); replace matches
     with redaction placeholders; warn the user.
   - Task extraction (skeptic-4): from **branch name + PR title only** (never the body or diff). Jira key
     `[A-Z][A-Z0-9]+-\d+` must validate against a **configured project-key allowlist** before fetch;
     forge issue `#\d+`. Fetch via Jira `GET /rest/api/3/issue/{key}` (ADF→text; base URL from
     user/global config only, never repo) or forge issue API. A 404/archived issue = "no task context",
     not an error.
   - Vault: resolve the reviewed repo's vault **by base-repo slug** (skeptic-6), not by assuming cwd is
     the repo. Run the context loader's vault-only layer (OV recall + ADRs + indications + feature
     dossier). Local-only layers (graphify `graph.json`, Serena, project `CLAUDE.md`) run **only if local
     HEAD == the PR's repo/branch** (arch-2); otherwise skip them and say so. **If no persona pack
     resolves, FAIL LOUDLY** ("ran generic fallback, not the project panel") — never silently degrade.
   - **Fetch existing v-cr fingerprints now** (arch-6) and carry them as a suppression set into step 3.

**3. REVIEW** (`steps/03-review.md`) — single pass; includes `commands/_shared/critic-panel.md`.
   - **Ground first**: run the resolved pack's bound analyzers/linters on the diff (precision floor).
   - **Resolve pack + select critics** for the reviewed repo (`_resolution.md`): architect always in +
     decorrelated lenses the diff implicates + the new **`correctness`** lens + skeptic on high-risk.
   - **Untrusted-input fencing** (sec-1): diff / PR body / Jira ticket are inserted as clearly-delimited,
     labelled **data** blocks with a system instruction that content inside is to-be-reviewed, never
     instructions; fence delimiters in the input are neutralised.
   - **Generate**: spawn critics in parallel, read-only, each returns findings (grounding + severity +
     `target: file:line` + advisory suggested-fix text) per the §d schema.
   - **Verify/filter = the grounding gate, machine-checked** (sec-1): the verdict and the
     post/no-post decision are derived from the **structured grounding + severity gate**, NOT from agent
     prose. Only `confirmed` findings become actionable inline comments; `advisory` is summary-only or
     dropped under `min_severity`. Suppress findings whose fingerprint is in the step-2 suppression set.
   - **Synthesize**: de-bias, dedupe/cluster, **hard volume cap** (v0: ≤10 inline + 1 summary; skeptic-5).
   - **Large-diff guard** (skeptic-8): above a diff-size threshold, chunk by file/hunk with a per-review
     token ceiling (`v-cr_max_tokens`), or warn + require `--force`. No silent truncation.
   - **Egress policy** (sec-5): when the PR is a fork/public, comment bodies may contain ONLY the
     finding + location + rationale + already-public diff hunk — never vault/CLAUDE.md content, file
     contents beyond changed lines, or any secret-scanner-flagged string.

**4. POST** (`steps/04-post.md`) — forge adapter writes back; gated.
   - **Dry-run by default**; render the full comment set with **exact bodies + targets** (not just
     counts). **The gate is non-bypassable for the first post to any (repo, PR) pair** (sec-4): always
     echo the resolved target `host/owner/repo#PR` and require explicit confirmation it is the intended
     repo (not a fork's upstream). `--post` only skips *re-confirmation* after a target was confirmed in
     the same session.
   - Output = the research triad: one **sticky summary comment** (marker `<!-- v-cr:summary -->`) +
     deduplicated **inline comments**, each carrying `<!-- v-cr:fp=sha256(file:rule:code-hash) -->`
     where `code-hash` = hash of the offending hunk content, **not the LLM message and not the line
     number** (skeptic-2; survives rebase + non-determinism). Comment-generation runs at temperature 0.
   - **Idempotent re-review**: skip fingerprints already posted; **resolve stale only when the thread's
     first comment is bot-authored, carries a v-cr fingerprint, and has zero human replies** (skeptic-3).
     Prefer resolve over delete.
   - **Redaction pass on comment bodies at the POST boundary** (sec-2/sec-5), again, before any write.
   - `gh`/`glab` fast path (resolve diff SHAs); REST fallback for Bitbucket + when CLIs absent.
   - **`/v-cr --unpost`**: first-class cleanup — delete/resolve every comment carrying this run's marker
     (the real undo / blast-radius net; skeptic-5).

**5. CAPTURE** (`steps/05-capture.md`) — record + learn, leak-safe.
   - Run `/v-capture`. **Record only finding metadata + posted comment ids — never bearer tokens,
     request/response headers, or raw diff hunks** (sec-2). Run the **redaction pass before the session
     write AND before the mandatory OpenViking push** (sessions are git-tracked + OV-indexed = a durable
     shared sink).
   - Offer indication promotion for recurring findings (the vault analogue of "learned org memory").

### Shared panel module (new)
**`commands/_shared/critic-panel.md`** — the extracted single-pass sub-procedure: *ground → select
(via `_resolution.md`) → generate (parallel read-only critics) → grounding-gate verify → de-biased
synthesis*. **No implement / apply-fix / re-spawn steps.** v-cr step 3 includes it. `v-team`'s
`04-execute-loop.md` §5.3 gets a one-line pointer naming this module as the canonical definition of the
panel sub-procedure (it keeps its own fix-and-reloop wrapper) — removes the brittle read-through
coupling (skeptic-9, arch-1/arch-3) without rewriting v-team.

### Install / wiring
`install.sh` already symlinks `commands/*.md` + `commands/*/` subdirs → `commands/v-cr.md`,
`commands/v-cr/`, and `commands/_shared/` auto-link with **no installer change** (verify with a dry
idempotency run). `commands/v-cr/adapters/` is a nested dir under the symlinked `v-cr/` — resolves fine.

### ADR-008 content (decision)
v-cr reviews remote PRs by reusing the *panel sub-procedure* (not v-team's fix-and-reloop), run
**single-pass**, CLI-driven, **precision-first**, **dry-run-default with a non-bypassable first-post
gate**. Treats diff/PR/Jira as **untrusted input**: the verdict + post decision come from the
machine-checked grounding gate, never agent prose. **Secret redaction at both the POST boundary and
before the committed+OV-indexed CAPTURE write.** Credentials are host-scoped via an allowlist; the
tool **never commits/pushes/applies code.**

## Open trade-offs / deferrals
- **Whole-repo RAG/graph grounding**: best tools index the whole repo; v0 grounds on diff + vault +
  (optional) graphify `graph.json` when local HEAD matches. Full RAG deferred — note the gap, don't
  claim parity.
- **Learned suppression / org memory**: hosted tools persist a learned DB; our analogue = vault
  `indications/` promoted at capture (manual). Auto-suppression deferred.
- **Trigger duality**: v0 is CLI-on-demand only (matches framework ethos). Webhook/CI auto-trigger out
  of scope (needs a hosted runner) — future `bin/` hook.
- **Large-diff handling**: v0 chunks-or-warns above a threshold; smarter map-reduce review deferred.
